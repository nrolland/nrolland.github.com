A slice of data abstraction

In the continuity of the talk given in Tokyo last month, I would like to lay down on F# code the concept explained. The theory is well explained in the paper Cardelli and Wegner-1985 from which those example come from.

Data abstraction, that is the precise control of what is allowed to be seen, is a fundamental aspect in programming, which rightly desserves some thoughts. And this is another place where types can help with enforcing the rules layed down.

Your first F# structure

Let's say you would like to implement a stack for integers. You could use the following code in F# to do so (style explicitely verbose)

module IntListStackModule = 
    let emptyStack : List<int>                    = List.empty
    let push       : int * List<int> -> List<int> = fun (e,l) -> List.Cons(e,l)
    let pop        : List<int> -> List<int>       = fun l -> List.tail l
    let top        : List<int> -> int             = fun l -> List.head l

Here the implementation is accessible to the consumer of the stack : it is a List. That means some potential misuse like the client making the assumption that your stack will always be a List, or modifing what you supposed to be only touched by you !

Going back to the spirit of the paper mentioned, let's be explicit about the types and use a record instead of a F# module

type IntListStack = {emptyStack : List<int>
                     push : int * List<int> -> List<int>
                     pop : List<int> -> List<int>
                     top : List<int> -> int }

let intListStack = { emptyStack  = List.empty
                     push        = List.Cons 
                     pop         = List.tail 
                     top         = List.head   }

You could also have an implementation based on array

type IntArrayStack = {emptyStack : int array * int
                      push : int *  (int array * int) -> int array * int
                      pop  : int array * int -> int array * int
                      top  : int array * int -> int }

let intArrayStack = { emptyStack  = Array.create 100 0, -1;
                      push        = fun (el,(a,i)) -> a.[i+1] <- el; (a,i+1);
                      pop         = fun (a,i)      -> (a,i-1);
                      top         = fun (a,i)      -> a.[i+1] }

Conversely the usual fold acts first on the last inserted element, aka the head of a list. It applies the reduce function, then recurses, as can be seen here :

let rec fold reduce zero l = match l with 
                             | x::xs -> fold reduce (reduce zero x) xs
                             | [] -> zero

let filterOdd' res x = filterOdd x res  
let r2 = fold filterOdd' [] [1;2;3;4;5] //=  [5; 3; 1] the order is NOT preserved !

It is a canonical and instructive exemple of continuation passing style to go from one to the other. Instead of using a foldback to be applied to the element directly, we can build up a function which, when called, will push its argument to the innermost

let fold' reduce zero alist = 
    let step x k = fun res -> k(reduce res x)
    (foldBack step alist id) zero

Let's re-read this definition : we apply a foldBack on the list alist at the last line. It will need a reducer which accepts the element of the list, x, some state, in our case a continuation k, and build a new state. In our case, the new state will be a new continuation, which takes some argument named 'res' in, asks the original reduce function to execute first, then hands over the control to the local continuation k which we got from the previous step.

So in a list [1,2,3], the last element x in the foldBack order, zero becomes the inception, and 1 will appear first to be called, as can be seen here

let f = fun s x-> printf "%A" x; ()
let step x k = fun res -> k(f res x)

let works = 
            fold' f () [1;2;3] = (foldBack step [1;2;3] id) ()                                                       &&
            fold' f () [1;2;3] = (step 1 (foldBack step [2;3] id)) ()                                               &&
            fold' f () [1;2;3] = (step 1 (step 2 (foldBack step [3] id))) ()                             &&
            fold' f () [1;2;3] = (step 1 (step 2 (step 3 (foldBack step [] id)))) ()                     &&
            fold' f () [1;2;3] = (step 1 (step 2 (step 3  id                  ))) ()                         &&
            fold' f () [1;2;3] = (step 1 (step 2 (fun zero3 -> id(f zero3 3  )))) ()                     &&
            fold' f () [1;2;3] = (step 1 (fun zero2 -> (fun zero3 -> id(f zero3 3))(f zero2 2))) ()          &&
            fold' f () [1;2;3] = (step 1 (fun zero2 -> f (f zero2 2) 3)) ()                                  &&
            fold' f () [1;2;3] = (fun zero1 -> f (f (f zero1 1) 2) 3) ()                                     &&
            fold' f () [1;2;3] = f (f (f () 1) 2) 3

The structure preservation of the right fold is a central property which has interesting consequences and can be generalized to other recursive types beyond lists.

cf the seminal paper by Meijer, Fokkinga, Paterson Functional Programming with Bananas, Lenses, Envelopes and Barbed Wire
